    Java基础

    String、StringBuffer 和 StringBuilder 的区别是什么? String 为什么是不可变的?

    String为不可变的对象，每次操作都会生成新的对象；StringBuilder和StringBuffer是可变的对象，其中StringBuilder线程不安全，StringBuffer线程安全。

    dasd

    因为String的底层是一个字节数组，被final修饰，数组一旦创建长度不可变，且由于final修饰，数组的引用不可指向其他对象，因此String不可变。

    StringBuffer/StringBuilder内部实际上是一个字节数组，这个字节数组没有被final修饰，StringBuffer和StringBulider的初始化容量应该为16，当存满之后会进行扩容，底层调用了数组拷贝的方法：System.arraycopy()…扩容的，所以StringBuffer/StringBuilder适用于字符串的频繁拼接操作。

    == 与 equals?hashCode 与 equals ?

    ==和equal都是比较相等的，其中==可以用作基本数据类型和引用数据类型，而equal只可以用作引用数据类型。

    ==在基本数据类型时比较的是值，引用数据类型比较的是内存地址。而equal方法的实现也是利用了==，如Object的equal方法。所以比较的是引用地址。但是有些类重写了equal方法，比较的是对象内的值，如包装类和String类。

    hashCode与equals都是为了比较对象是否相等一致的，hashcode只要比较的是对象的哈希值就可以初步判断对象是否相同，效率比较高，但是不同对象的hash值可能相同，因此还需要用equal方法进一步判断。Java是这样规定的：

    • equal()相等的两个对象他们的hashCode()肯定相等
    • hashCode()相等的两个对象他们的equal()不一定相等

    Java中的集合类往往是根据先判断hashCode，如果相同再调用equal方法进行判断。

    Java 反射？反射有什么缺点？你是怎么理解反射的（为什么框架需要反射）？

    在程序运行期间，对于任何一个类，能获取到这个类的所有方法和属性，还能调用这些方法和属性，这种功能称为反射。

    反射的应用场景了解么？

    框架中也大量使用了动态代理，而动态代理的实现也依赖反射。

    反射机制的优缺点

    优点 ： 可以让咱们的代码更加灵活

    缺点 ：增加了安全问题。比如可以强制获取private属性比如可以无视泛型参数的安全检查。

    BIO,NIO,AIO 有什么区别?

    BIO (Blocking I/O): 同步阻塞 I/O 模式，数据的读取写入必须阻塞在一个线程内等待其完成。

    NIO (Non-blocking/New I/O):是一种同步非阻塞的 I/O 模型，它支持面向缓冲的，基于通道的 I/O 操作方法。

    AIO (Asynchronous I/O):异步非阻塞的 IO 模型。异步 IO 是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会堵塞在那里，当后台处理完成，操作系统会通知相应的线程进行后续的操作。

    写时复制

    CopyOnWrite容器即写时复制的容器。往一个容器添加元素的时候，不直接往当前容器Object[]添加，而是先将当前容器Object[]进行Copy，复制出一个新的容器Object[] newElements，然后向新的容器Object[] newElements里添加元素。添加元素后，再将原容器的引用指向新的容器setArray(newElements)。这样做的好处是可以对CopyOnWrite容器进行并发的读，而不需要加锁，因为当前容器不会添加任何元素。所以CopyOnWrite容器也是一种读写分离的思想，读和写不同的容器。

    比较 HashSet、LinkedHashSet 和 TreeSet 三者的异同

    hashSet是基于hash算法实现，查找很快，但无法控制元素顺序。LinkedHashSet元素是hashSet的子类，在hashSet的基础上有一个双向链表，元素有序。TreeSet的底层基于红黑树实现的，元素有序。

    HashMap 和 HashSet 区别？

    HashSet是通过HasMap来实现的，HashMap的输入参数有Key、Value两个组成，在实现HashSet的时候，保持HashMap的Value为常量，相当于在HashMap中只对Key对象进行处理。

    HashMap 和 TreeMap 区别？

    HashMap:

    允许null作为key和value,不保证元素迭代顺序是按照插入时的顺序

    TreeMap:
    基于红黑树实现，不允许null，存入TreeMap的元素应当实现Comparable接口或者实现Comparator接口，会按照排序后的顺序迭代元素。

    JVM

    jvm内存结构

    分为线程公有和线程私有

    线程公有

    堆，虚拟机启动时创建，作用：存放实例对象，几乎所有的对象和数组都在这里分配内存。

    方法区：存储虚拟机加载的类学习，常量、静态变量。jdk1.7方法区是实现是永久代，jdk1.8后是元空间，且移动到了直接内存中。

    线程私有

    程序计数器：当前线程执行字节码的行号指示器，用来选取下一条指令。

    虚拟机栈：堆管存储，栈管运行；虚拟机栈就是用来存储方法运行时的

    局部变量表，操作数栈等运行数据。存储的数据类型有八大基本数据类型和引用类型。

    本地方法栈：和虚拟机栈类似，不过是为本地方法服务的。

    什么是类加载？何时类加载？类加载流程？

    通过类的全限定名把类的class文件加载进JVM内存，并把类的信息转化为JVM方法区中的运行时数据的过程。

    类加载时机

    1、当jvm执行new指令时会初始化类。即当程序创建一个类的实例对象

    2、即程序访问类的静态变量(不是静态常量，常量会被加载到运行时常量池)，或为静态变量赋值，或调用静态方法时。

    3、使用 java.lang.reflect 包的方法对类进行反射调用时Class.forname(“…”), newInstance()等等。如果类没初始化，需要触发其初始化。

    4、初始化一个类，如果其父类还未初始化，则先触发该父类的初始化。

    5、当虚拟机启动时，用户需要定义一个要执行的主类 (包含 main 方法的那个类)，虚拟机会先初始化这个类。

    类加载流程

    1、加载 ，通过全类名获取class文件，并把它加载进JVM中，在方法区生成对应的运行时数据结构，还会在内存中生成class对象。

    2、连接，包括验证和为类变量进行初始化。

    3、初始化，执行类的初始化方法，。

    GC 的全流程

    1：eden、SurvivorFrom 复制到 SurvivorTo，年龄+1
    首先，当Eden区满的时候会触发第一次GC,把还活着的对象拷贝到SurvivorFrom区，当Eden区再次触发GC的时候会扫描Eden区和From区域,对这两个区域进行垃圾回收，经过这次回收后还存活的对象,则直接复制到To区域（如果有对象的年龄已经达到了老年的标准，则赋值到老年代区），同时把这些对象的年龄+1

    2：清空 eden、SurvivorFrom ，然后，清空Eden和SurvivorFrom中的对象，也即复制之后有交换，谁空谁是to

    3：SurvivorTo和 SurvivorFrom 互换
    最后，SurvivorTo和SurvivorFrom互换，原SurvivorTo成为下一次GC时的SurvivorFrom区。部分对象会在From和To区域中复制来复制去,如此交换15次(由JVM参数MaxTenuringThreshold决定,这个参数默认是15),最终如果还是存活,就存入到老年代

    总结：

    大部分情况，对象都会首先在 Eden 区域分配，在一次新生代垃圾回收后，如果对象还存活，则会进入 s0 或者 s1，并且对象的年龄还会加 1(Eden 区->Survivor 区后对象的初始年龄变为 1)，当它的年龄增加到一定程度（默认为 15 岁），就会被晋升到老年代中。

    线程和进程的区别

    进程是程序的一次运行活动，它是操作系统资源分配的最小单位，一个进程可由一个或多个线程组成，线程可以利用进程的资源独立运行和调度，线程比进程更小，因此线程的开销比进程小，但同一个进程的不同线程之间存在线程安全问题，而不同进程之间是独立运行的。

    什么是上下文切换?

    多线程运行环境下，当前任务在执行完 CPU 时间片切换到另一个任务之前会先保存自己的状态，以便下次再切换回这个任务时，可以再加载这个任务的状态。任务从保存到再加载的过程就是一次上下文切换。

    什么是线程死锁?如何避免死锁?

    多个线程都在等待对方释放资源而阻塞，程序无法正常终止的情况。

    破坏请求和保持条件，线程可以在获取需要的资源后再启动，运行完成后马上释放。

    破坏不可剥夺条件，在等待获取资源超过一定时间还无法获取后，可以自动释放手中的资源。

    乐观锁和悲观锁了解么？

    乐观锁是一种乐观思想，假设当前时刻不会有线程跟它竞争资源，不会给上锁，适用于读多写少的场景。

    悲观锁是一种悲观思想，认为任何时刻都存在线程竞争关系，会上锁，适用于读少写多的场景。

    Java 线程池有哪些参数？阻塞队列有几种？拒绝策略有几种？新线程添加的流程?

    线程池特点：线程复用、控制最大并发数;管理线程。
    线程池的作用

    第一：降低资源消耗。通过线程复用降低线程创建和销毁造成的销耗。

    第二：提高响应速度。当任务到达时，任务可以不需要等待线程创建就能立即执行。

    第三：提高线程的可管理性。线程是稀缺资源，如果无限制的创建，不仅会销耗系统资源，还会降低系统的稳定性，使用线程池可以进行统一的分配，调优和监控。

    参数
    • 核心线程数：线程池常驻线程数
    • 最大线程数：线程池中最大可运行的线程数
    • 任务队列：存放被提交但尚未执行的任务
    • 存活时间：多余的空闲线程的存活时间，当前池中线程数量超过corePoolSize时，当空闲时间达到keepAliveTime时，多余线程会被销毁直到只剩下corePoolSize个线程为
    • 存活时间单位
    • 拒绝策略
    • threadFactory：表示生成线程池中工作线程的线程工厂，用于创建线程，一般默认的即可

    1、AbortPolicy(默认)：直接抛出RejectedExecutionException异常阻止系统正常运行

    2、DiscardOldestPolicy：抛弃队列中等待最久的任务，然后把当前任务加人队列中，尝试再次提交当前任务。

    3、DiscardPolicy：该策略默默地丢弃无法处理的任务，不予任何处理也不抛出异常。如果允许任务丢失，这是最好的一种策略。

    4、CallerRunsPolicy：“调用者运行”一种调节机制，该策略既不会抛弃任务，也不会抛出异常，而是将某些任务回退到调用者，从而降低新任务的流量。

    阻塞队列种类

    ArrayBlockingQueue：由数组结构组成的有界阻塞队列
    LinkedBlockingQueue：由链表结构组成的有界（但大小默认值为integer.MAX_VALUE）阻塞队列。
    SynchronousQueue：不存储元素的阻塞队列，也即单个元素的队列。
    PriorityBlockingQueue：支持优先级排序的无界阻塞队列。

    讲一下 JMM(Java 内存模型)。volatile 关键字解决了什么问题？说说 synchronized 关键字和 volatile 关键字的区别。

    JMM内存模型，不同线程有一块共享内存，不可以直接在上面操作，每个线程需要把数据复制到自己的用户内存中上操作，再把更改数据写回主内存。
    因为用户内存仅各自的线程可见，因此存在可见性问题，不同线程的操作还可能覆盖，还存在原子性问题。

    volatile解决了可见性和防止指令重排序问题。

    synchronized是用于线程同步的关键字，可作用在方法和代码块中，还可以解决原子性的问题。
    volatile是轻量级的锁，只能作用于变量。

    TCP 的三次握手与四次挥手的内容？ TCP 为什么连接是三次握手而断开是四次握手？

    三次握手
    • 客户端–发送带有 SYN 标志的请求数据包，同时选择一个初始序列号 x –一次握手–服务端
    • 服务端–发送带有 SYN/ACK 标志的响应数据包，确认号ack = x+1,同时为自己初始化一个序列号y–二次握手–客户端
    • 客户端收到确认后再给服务器发送一个带有ACK 标志的响应数据包，确认号ack = y+1;
    四次挥手
    • 客户端-发送一个 FIN，用来关闭客户端到服务器的数据传送
    • 服务器-收到这个 FIN，它发回一 个 ACK，确认序号为收到的序号加1 。
    • 服务器-关闭与客户端的连接，发送一个FIN给客户端
    • 客户端-发回 ACK 报文确认，并将确认序号设置为收到序号加1

    四次挥手中客户端在发送完确认序号后还要等待2 ∗MSL（最长报文段寿命）的时间后才会进入close状态。

    为什么客户端最后还要等待2MSL？

    保证客户端发送的最后一个ACK报文能够到达服务器，因为这个ACK报文可能丢失，如果丢失，服务端重发的FIN报文能在2MSL内到达客户端。

    TCP 为什么连接是三次握手而断开是四次握手？

    因为请求链接过程，ACK和SYN可以同时发送给客户端。而连接断开时，服务端可能还有一些数据要继续发给客户端，所以只能先发送ACK报文段，表示确认收到断开连接的请求，后面在发FIN信号。

    如果已经建立了连接，但是客户端突然出现故障了怎么办？

    2小时内服务端没有收到客户端任何的信息就会发送一个探测报文段，每隔75秒发送一次。若一连发送10个探测报文仍然没反应，服务器就认为客户端出了故障，接着就关闭连接。

    TCP 与 UDP 的区别及使用场景

    TCP

    提供可靠的，面向连接的数据服务。

    传送数据之前必须先建立连接，数据传送结束后要释放连接。

    确认、窗口、重传、拥塞控制机制保证数据的可靠传输

    传输方式：字节流，传输效率低

    应用场景：文件传输、邮件传输、远程登录

    UDP

    无连接，尽最大努力的数据服务

    发送方发送不需要建立连接，接收方接受不需要确认

    传输方式：数据报段，传输效率高

    应用场景：即时通信，QQ语言，视频直播

    TCP协议如何保证可靠传输

    1. TCP发送方为每个发送的数据块编号，接收方接受后会重排为有序的数据。
    2. 首部校验和，如果检验和有差错，TCP 将丢弃这个报文段和不确认收到此报文段。
    3. 接收方确认机制，发送方超时重传【ARQ协议】
    4. 流量控制与拥塞控制

    一次完整的 HTTP 请求所经的步骤

    客户端根据DNS协议解析域名，得到服务端的IP地址。
    客户端根据IP地址找到服务端，请求建立连接，TCP三次握手建立连接。
    服务端根据请求响应，并将响应结果返回给客户端。
    所有数据传输完成后，TCP四次挥手，连接断开。
